#1 Viewport Scroll Basics

Scrolling through this page demonstrates viewport scrolling. The fixed header above tracks your live position: scrollY, total pageHeight, percentage progress, and viewport height.

Use scroll(direction='down') to move through this page. Each call scrolls by roughly one viewport height. The return value includes the new scroll_y, page_height, viewport_height, and a boolean scrolled indicating whether position actually changed.

When no css_selector is given, the tool scrolls the entire page (viewport). When a selector is provided, it scrolls that specific container.

The behavior parameter controls whether scrolling is instant (default) or animated. This page has CSS scroll-behavior: smooth set — the default behavior='instant' overrides it for predictable automation. Use behavior='smooth' for visible animation in demos.

#2 Boundary Detection

When a scroll operation hits the top or bottom boundary, the tool returns scrolled: false. This tells the agent there is no more content in that direction — no need to keep scrolling.

Try scrolling up from the very top of this page. The tool returns {"scrolled": false, "scroll_y": 0, ...} because the viewport is already at position zero.

The same applies at the bottom: once scrollY + viewportHeight ≥ pageHeight, scrolling down returns scrolled: false.

This boundary detection works identically for container scrolls. A container at its bottom edge returns scrolled: false for direction='down'.

#3 Instant vs Smooth Scroll Comparison

The behavior parameter controls scroll animation. 'instant' (default) teleports immediately. 'smooth' uses the browser's native animation (~300–500ms) and waits for the scrollend event before returning position data. Both return the same final position — only the visual experience differs.

This page has scroll-behavior: smooth set in CSS. The default behavior='instant' overrides it. behavior='smooth' respects it.

Under the hood, smooth scrolling uses a three-signal waiting strategy: scrollend event (primary), double-rAF no-movement check (~32ms early exit for boundaries), and a 3-second timeout fallback.

#7 Scroll Into View — Centering

The target element below will be CENTERED in the viewport when scrolled into view, not pushed to the bottom edge.

Selenium's native scroll_to_element() aligns the element's bottom to the viewport's bottom, which often leaves the element barely visible. We use scrollIntoView({block: 'center'}) instead, placing the element in the middle of the visible area.

With behavior='smooth', scroll-into-view produces a visible animation as the page glides to center the element — ideal for product demos.

• Composable Patterns

These are not scrollable scenarios — they document how agents combine the scroll tool's return values to accomplish common tasks.

Scroll-to-Bottom

Use the returned page_height and viewport_height to calculate remaining distance. Keep calling scroll(direction='down') until scrolled: false.

Percentage Scrolling

scroll_y / page_height gives current progress as a fraction. The fixed header above shows this as a live percentage.

Infinite Scroll Scraping

Call scroll(direction='down') followed by wait_for_network_idle() in a loop. New content loads on each scroll. Stop when scrolled: false — the page has no more content to load.

Container Content Discovery

Scroll a container to its bottom with repeated scroll(direction='down', css_selector='...') calls. Read content at each step with get_page_text(selector='...'). The return value's scroll_height tells you the total scrollable content size.

Instant vs Smooth

Use behavior='instant' (default) for automation speed and predictable timing. Use behavior='smooth' for user-visible demos, recordings, and when matching the site's expected scroll experience.
